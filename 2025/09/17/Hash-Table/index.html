雜湊表(Hash Table)

想像一下，你要在一本厚厚的書裡找「第十章節的內容是什麼？」
如果書沒有目錄（就像一個普通的清單 Array）
你只能一頁一頁翻，直到翻到第十章節，這樣搜尋速度非常慢
當書只有 10 頁還好，但如果有 1000 頁，這就會變得很低效

但如果書有「目錄」，你只要先查目錄找到該章節的頁碼
就能直接翻到對應的頁數（像是直接定位到索引值 Index）
這樣搜尋就會變得非常快！

Hash Table就像是幫資料加上一個「目錄」，
讓你能快速跳到正確的位置，而不需要一個一個比對。

什麼是 Hash Table？

**Hash Table（雜湊表）**是一種用來快速存取資料的資料結構。
它的核心概念是：

  • 每一筆資料都有一個 Key（鑰匙）。
  • 透過 Hash Function（雜湊函數），這個 Key 會被轉換成一個「索引值」。
  • 這個索引值就告訴我們，資料要放在哪個位置（Bucket）。
    這樣一來，當我們想要存取某筆資料時，
    只要用同樣的 Key 再跑一次雜湊函數，就能 直接跳到正確的位置，幾乎不需要搜尋。

生活比喻

把 Hash Table 想像成一間超市：

  1. 你手上有一樣商品（Key），例如「牛奶」、「餅乾」或「牛肉」。
  2. 超市的系統會用「雜湊函數（Hash Function）」計算出它該放在哪個區域（Index）。
  3. 商品就會被放到對應的貨架區（Bucket），像是乳製品區、零食區或肉類區。

Hash Function（雜湊函數）

要讓 Hash Table 運作，最重要的就是 Hash Function（雜湊函數）
它是一個「公式」或「演算法」，用來把 Key 轉換成一個 索引值（Index）
決定資料要放在哪個位置。
Hash Function 應該具備：

  1. 計算速度快 → 不能太複雜
  2. 分布平均 → 資料不要都集中在某些位置，否則容易發生衝突
  3. 相同的 Key 必須得到相同的結果 → 才能正確找到資料

常見的 Hash Function

以下是一些常見的雜湊函數設計方法：

  1. 除法取餘數法（Division Method）

    就像超市有 m 個分區，依照「商品編號 % m」來決定放哪裡
    例子：商品編號 12345 → 12345 % 10 = 5 → 放到第 5 區

  2. 乘法法（Multiplication Method）

    想像店員用一個隱藏的比例常數 A，先把商品編號乘上 A，
    取出小數部分，再放大到超市分區數 m，決定放到哪個區域
    例子：商品編號 12345，計算後可能被分到第 7 區

  3. 字串雜湊（String Hashing）

    當商品名字是字串（例如「牛奶」、「餅乾」），
    我們先把每個字元轉成數字，再依照公式算出在哪個分區。
    例子：”abc” → 97p^0 + 98p^1 + 99*p^2 → 對 m 取餘數 → 分到某個區域。

碰撞（Collision）

雖然 Hash Function 能幫我們把資料分配到不同位置，
但有時候 不同的 Key 可能算出相同的索引值，這種情況就叫做 碰撞（Collision）。

為什麼會碰撞？

舉例來說：

  • 我們的公式是 index = 學號 % 10
  • 學號 12345 % 10 = 5
  • 學號 54325 % 10 = 5

這兩筆資料都要放到第 5 號位置，這就是碰撞。

常見解決方法

1. 鏈結法（Chaining）

Source：Pumpkin Programmer

  • 每個位置（Bucket）不只放一筆資料，而是放一個「鏈結串列」。
  • 碰撞時，把新的資料加到串列裡。

優點：簡單易懂，不怕資料增加。
缺點：如果太多碰撞，搜尋效率會變慢。

2. 開放定址法（Open Addressing）

Source：Open Addressing

  • 如果原本的位置被佔了，就找下一個空位放。
  • 常見策略：
    • 線性探測（Linear Probing）：往後一格一格找。
    • 二次探測（Quadratic Probing）：往後跳 1²、2²、3²… 去找。
    • 雙重雜湊（Double Hashing）：再用另一個 Hash Function 幫忙。

優點：不需要額外鏈結結構。
缺點：如果裝太滿，碰撞會更多，效率會下降。

Hash Table 的優缺點

雖然 Hash Table 很強大，但它並不是完美的。
我們來看看它的 優點 和 缺點：

優點 👍

  1. 存取速度快

    • 插入、查找、刪除的平均時間複雜度都是 O(1)。
    • 適合需要大量快速查詢的應用場景。

  2. 使用簡單

    • 大部分程式語言都已經幫你實作好 Hash Table。
    • 例如：
      • Python → dict/ set
      • Java → HashMap / HashSet
      • JavaScript → Object / Map

缺點 👎

  1. 碰撞（Collision）問題

    • 不同的 Key 可能會被分配到同一個位置。
    • 如果碰撞太多，效率就會下降。

  2. 需要好的 Hash Function

    • Hash Function 設計不好，資料會集中在少數位置，導致效能差。

  3. 記憶體消耗較大

    • 為了減少碰撞，Hash Table 通常會配置比實際資料量更多的空間。

  4. 元素無法保證順序

    • Hash Table 注重「快」而不是「順序」。
    • 如果你需要保持資料的順序，Hash Table 並不適合。
